## 设计原则

### 1. 音乐播放器只修正布局，不回退功能
本次不删除已新增的：
- 播放模式切换
- 下一首按钮
- 收起按钮
- 展开容器内进度条

只调整：
- 展开态内容分区
- 高度计算
- 内边距与块间距
- 按钮和歌单的排列方式

### 2. “原版”是新增 preset，不覆盖现有风格
用户要求是：
- 新增真正原版
- 不移除当前“经典 / 精致”

因此不采用“替换 classic”的方式，而是采用第三个 preset：
- `original`

### 3. 卡片风格仍通过全局变量集中管理
继续沿用当前架构：
- 在根节点设置 `data-card-style`
- 在 `globals.css` 中为各 preset 提供变量
- `.card`、`.card-rounded` 等公共类自动继承
- 仅对少量局部写死样式做补充适配

### 4. 只在当前 preset 机制的作用边界内做“完整原版”
“完整原版”在本次实现中的含义是：
- 不只是首页主卡片容器
- 还包括依赖 card 变量的局部圆角、边框、模糊、阴影表现

但不追求：
- 对所有组件进行旧版 DOM 逐像素复刻
- 重写与 preset 无关的组件内部结构

也就是说：
- **恢复旧版 card 基础视觉语言**
- **让现有 preset 机制能稳定承载这套风格**
- **避免把范围扩展成整站 UI 重构**

## 音乐播放器展开态设计

### 根因分析
当前展开态的问题来自两层不匹配：

1. **内容块变多**
   现在展开态实际包含：
   - 顶部主行（图标、标题、播放按钮）
   - 进度条块
   - 控制按钮行
   - 歌单列表

2. **展开高度估算过于保守**
   当前高度公式不足以覆盖：
   - 外层 `p-4`
   - 进度条块自身高度
   - 控制按钮行高度
   - 各块之间的 gap
   - 最底部缓冲空间

因此即使 JSX 结构本身成立，也会出现截图里这种“内容被压缩”的问题。

### 目标布局
展开态建议拆成 3 个垂直区块：

#### 区块 1：头部主行
保留：
- 音乐图标
- 当前曲名
- 播放/暂停按钮

职责：
- 只承担主信息与主播放控制
- 不再把展开态进度条塞在这一行内部

#### 区块 2：展开态控制区
单独放入展开容器内部，包含：
- 进度条
- 当前时间 / 总时长
- 播放模式切换
- 下一首
- 收起按钮

原则：
- 进度条属于展开区，不属于顶部切换区
- 控件之间有稳定间距
- 所有控制点击都阻止冒泡，避免误触展开/收起

#### 区块 3：歌单列表
保留现有歌单项结构，但保证：
- 与控制区之间有明确间距
- 与底边之间有明确留白
- 最后一项不会贴底

### 高度计算策略
不再沿用当前“基础高度 + 歌单项高度 + 一个经验常数”的粗糙方式，而是让高度与结构对应。

建议思路：
- 基础头部高度
- 进度条块高度
- 控制行高度
- 歌单项高度 × 歌曲数
- 顶底 padding
- 区块间 gap
- 额外底部缓冲

这样做的目的不是“动态测量”，而是让公式与真实布局一致。

### 边界场景约束
本次设计还需要明确以下边界行为：

1. **当前默认歌单长度（3 首）**
   - 必须完整展示
   - 最后一项与容器底边保留明显留白

2. **窄宽度或字体略放大时**
   - 顶部主行不能把播放按钮挤出容器
   - 必要时优先增加纵向留白，而不是继续压缩按钮区

3. **未来歌单项增加时**
   - 如果展开高度超过安全显示范围，优先让歌单列表区可滚动
   - 不允许继续无限增高到破坏底部定位逻辑

### 交互约束
- 展开态进度条点击：
  - 只影响 seek
  - 不触发收起/展开
- 控制按钮：
  - 继续使用 `stopPropagation`
- 歌单项点击：
  - 继续切歌
  - 行为不变

### 预期结果
修复后应达到：
- 展开态观感不拥挤
- 播放按钮不再顶边
- 最后一首歌不贴底
- 展开态进度条可正常点击/跳转
- 保留现有新增功能

## 原版卡片风格设计

### 数据结构
当前字段在 `siteContent.theme.cardStylePreset` 下。

本次扩展为：

```ts
cardStylePreset: 'original' | 'classic' | 'refined'
```

### 默认值
用户已要求默认切换到原版，因此：

- `src/config/site-content.json` 默认值改为 `original`
- 运行时 fallback 从 `classic` 改为 `original`
- **缺失字段或非法值**（例如手动写入未知字符串）都统一回落到 `original`

这包括：
- `src/app/layout.tsx`
- `src/app/(home)/config-dialog/index.tsx`
- `src/app/(home)/config-dialog/site-settings/index.tsx`
- 以及其他读取该字段并提供兜底的地方

### 原版视觉来源
以旧仓库 `blog-elaina-cloudflare-前端优化前` 为准，核心视觉参数直接参考：
- `/app/blog-elaina-cloudflare-前端优化前/src/styles/globals.css:94-100`
- `/app/blog-elaina-cloudflare-前端优化前/src/styles/globals.css:109-114`

旧版 `.card` 的关键样式为：
- `rounded-[40px]`
- `border`
- `bg-card`
- `box-shadow: 0 40px 50px -32px rgba(0, 0, 0, 0.05), inset 0 0 20px rgba(255, 255, 255, 0.25)`
- `backdrop-filter: blur(4px)`

因此本次 `original` 的最小强约束不是“看起来接近”，而是以下 token 必须与旧版保持一致或等价：
- 外圆角：`40px`
- 公共 card 背景：`var(--color-card)`
- 边框：保留旧版亮白边框语义，默认使用 `rgba(255, 255, 255, 0.9)`
- 外阴影：`0 40px 50px -32px rgba(0, 0, 0, 0.05)`
- 内阴影：`inset 0 0 20px rgba(255, 255, 255, 0.25)`
- 毛玻璃 blur：`4px`

允许实现层做的自由度仅限于：
- 是否继续通过 CSS 变量承载这些值
- `card-rounded` 内圆角是否使用与旧版等价的映射方式

不允许的偏离：
- 用当前 `classic` 的参数直接改名为 `original`
- 只保留“大概相似”的视觉方向但不对齐上述核心值

### preset 定义

#### `original`
目标：尽量回到旧仓库最初 card 观感。

重点特征：
- 大圆角
- 更亮边框
- 轻 blur
- 接近旧版的阴影层次
- `bg-card` 主导的卡片背景

建议参数方向：
- `--card-radius: 40px`
- `--card-inner-radius: 40px` 或与旧版局部圆角更一致的值
- `--card-border-color: rgba(255, 255, 255, 0.9)`
- `--card-shadow: 0 40px 50px -32px rgba(0, 0, 0, 0.05), inset 0 0 20px rgba(255,255,255,0.25)`
- `--card-blur: 4px`
- `--card-bg: var(--color-card)`

#### `classic`
目标：保留当前 classic，不改语义。

#### `refined`
目标：保留当前 refined，不改语义。

### 为什么要保留三套
因为它们表达的是三个不同层级：

- `original`：真正旧仓库原版观感
- `classic`：当前仓库里较熟悉、较稳妥的中间版本
- `refined`：更收敛、更现代的版本

这样既满足用户要“原版”，也不破坏当前已有的两套 preset。

## 风险与应对

### 风险 1：音乐卡片高度修复不彻底
如果只是微调 margin/padding，而不调整高度公式，可能仍会出现：
- 按钮贴边
- 最后一项贴底

**应对：**
- 高度公式必须与区块结构对应
- 不能只做样式微调

### 风险 2：原版 preset 只是“看起来像 classic”
如果只是复制当前 classic 并改名字，会违背用户目标。

**应对：**
- 以旧仓库 `前端优化前` 的 `.card` 为直接参考
- 至少在圆角、边框、阴影、blur、背景这几个核心参数上明确拉开差异

### 风险 3：默认值遗漏
如果只改 `site-content.json` 而不改 fallback，旧配置或预览态仍会落回 `classic`。

**应对：**
- 全局搜索所有 `cardStylePreset ?? 'classic'`
- 统一替换为 `original`

### 风险 4：预览与保存不一致
如果设置 UI 增加了“原版”按钮，但类型或保存逻辑未同步，可能出现：
- 预览可切换
- 保存失败或保存后回退

**应对：**
- 类型、UI、运行时 fallback、保存链路一起修改
