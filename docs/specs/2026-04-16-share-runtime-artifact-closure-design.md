# /share 正式产物闭环与双栏导航设计

## 目标

在不把 `/share` 切换为直接读取数据库的前提下，把 `/share` 从“已切到正式产物”补齐到“运行时正式产物 + 编辑态完整字段 + 保存链路一致闭环”的可验收状态。

本阶段聚焦：
- `/share` 运行时继续消费 `public/share/*` 正式产物
- `/share` 页面引入双栏导航：左侧目录树主导，顶部分类 tabs 在当前目录内进一步筛选
- 编辑态把 `category` 与 `folderPath` 变成真实可维护字段
- 保存 / 删除 / URL 修改后，`public/share/list.json`、`categories.json`、`folders.json`、`storage.json` 一起更新
- 保持现有 `search + tags` 过滤能力
- 通过 `pnpm run dev` 查看真实页面效果，而不是额外引入独立原型工具

本阶段**不包含**：
- 为 `/share` 新增 blog 对等级别的 preview / execute / verify / rebuild 开发工具面板
- 把 `/share` 运行时切换为直接读取 `content.db`
- 额外扩展 `/share` 的新产品能力（例如收藏系统、排序规则大改、批量操作后台等）

## 当前事实与收口方向

### 当前事实
- `/share` 页面当前主要读 `public/share/list.json`
- 保存链路已经会写：
  - `public/share/list.json`
  - `public/share/categories.json`
  - `public/share/folders.json`
  - `public/share/storage.json`
- `share-storage` / `share-artifacts` 已有基础能力，但 UI 目前没有真正消费目录树与分类产物
- 编辑态当前只维护：名称、logo、url、description、tags、stars
- `category` / `folderPath` 在存储模型里已存在，但尚未成为产品层正式能力

### 本阶段收口方向
把 `category` / `folderPath` 从“存储里有字段”推进成：
- 页面真实消费
- 编辑态真实维护
- 正式产物真实更新
- 文档真实描述

## 页面信息架构

### 总体结构
`/share` 升级为双栏导航内容页，但视觉上仍保持“分享资源页”的轻量感，不做成后台管理界面。

- 左侧：目录树
- 右侧顶部：分类 tabs
- 右侧次级过滤：搜索框 + 标签筛选
- 右侧主体：share 卡片网格

### 左侧目录树
- 目录树是主导航
- 有“全部目录”节点
- 支持层级目录
- 点击某个目录后，右侧内容范围切到该目录及其子目录下的 share
- 父目录本身也是有效过滤节点
- 前台默认不展示没有任何 share 的空目录节点

### 右侧分类 tabs
- 只在当前目录范围内生效
- 永远包含“全部”
- 只展示当前目录范围内实际存在内容的分类
- 切换目录时：
  - 若当前分类在新目录范围内仍有内容，则保留
  - 否则自动回到“全部”

### 搜索框与标签筛选
- 继续保留当前页面已有的搜索框与标签筛选
- 角色从“主过滤器”降为“细粒度筛选器”
- 不承担内容组织结构职责

### 卡片区域
- 继续沿用现有 share card 风格
- 不引入完全不同的视觉系统
- 编辑态与浏览态共享同一页面上下文

## 过滤顺序与交互语义

固定过滤顺序为：

`目录 -> 分类 -> 搜索 -> 标签`

### 语义说明
1. 目录先决定当前内容空间
2. 分类在当前目录范围内做细分
3. 搜索进一步缩小当前结果集
4. 标签做最后一层细筛

### 保留状态策略
- 切换目录时，若当前分类在新目录下无结果，则回退到“全部”
- 切换目录或分类时，保留搜索词
- 切换目录或分类时，保留标签筛选
- 但空状态必须明确说明是：
  - 当前目录为空
  - 当前目录下该分类无结果
  - 搜索 / 标签把结果筛空

### 编辑态与浏览态关系
- 进入编辑态后，不改变当前浏览上下文
- 在某目录下编辑或新建 share，保存后如果该条目仍符合当前过滤条件，则继续留在当前视图中
- 若保存后不再符合当前过滤条件，则允许从当前视图中消失，这属于合理结果

## 编辑态字段设计

### 保留字段
- `name`
- `logo`
- `url`
- `description`
- `tags`
- `stars`

### 新增正式字段
- `category`
- `folderPath`

### 分类维护方式
- 分类允许直接输入或修改
- 可以提供已有分类建议，但不强制只能选已有值
- 分类保持轻量，不承担层级结构职责
- 分类不是必填项
- 空字符串或仅空白输入会被归一化为“无分类”
- 存储与导出前会做 `trim()` 归一化，但不在本阶段额外引入大小写统一或别名合并规则

### 目录维护方式
- 目录不使用纯文本自由输入
- 使用目录选择器，并允许即时新建目录
- 新建目录后应立即加入当前可选项并自动选中
- 即使正式产物尚未刷新，当前表单也必须保留这个新目录值
- `folderPath` 不是必填项
- 空值表示“未归档 / 不属于任何目录”
- `folderPath` 存储规则与 blog 保持一致：
  - 先按 `/` 切分路径片段
  - 每段做 `trim()`
  - 过滤空片段
  - 重新编码为以 `/` 开头的规范路径，例如 `/设计/图片工具`
- 不使用单独的根节点值；“全部目录”只存在于 UI 过滤层，不写入正式产物

该行为应尽量复用 blog 已验证过的目录选择思路，而不是重新发明一套完全不同的目录交互。

### 正式产物字段契约
- `list.json` 是 `/share` 页面卡片列表的直接运行时来源
- `list.json` 中每条 share 继续保留运行时直接需要消费的列表字段，并在本阶段显式包含：
  - `category`
  - `folderPath`
- `/share` 页面中的目录过滤、分类过滤、搜索与标签叠加过滤，均基于 `list.json` 中的条目字段完成
- `categories.json` 为 `{ categories: string[] }`，用于生成顶部分类 tabs 的候选集合
- `folders.json` 继续沿用现有目录树节点结构，节点路径使用规范化后的 `folderPath`，用于生成左侧目录树
- `storage.json` 中：
  - `category` 缺失表示无分类
  - `folderPath` 缺失表示未归档
  - 不写入空字符串分类或空字符串目录路径
- 历史数据中的空白分类、空白目录路径与不规范目录路径，在本阶段保存/导出时一并归一化
- “无分类”与“未归档”在数据层都表示为缺失字段，而不是写入字面量值
- 前台不额外提供“无分类”tab 或“未归档目录”节点；这类条目只在“全部目录”或对应目录上下文里的综合结果中出现

### 排序规则
- `categories.json` 中的分类按字符串升序稳定输出
- `folders.json` 沿用现有目录树稳定排序规则
- `list.json` 继续保持现有 share 导出顺序语义；本阶段不额外引入新的排序系统

### URL 冲突策略
- URL 是本阶段已支持的稳定更新锚点，也是 published share 的唯一业务标识
- 新建 share 时，如果目标 URL 已存在于某条 published 记录中，则必须阻止保存并返回显式错误，不允许静默覆盖
- 编辑既有 share 时：
  - 若 URL 未变化，则按同一记录更新
  - 若 URL 修改为一个新的、未被 published 记录占用的 URL，则允许更新
  - 若 URL 修改为另一个已被 published 记录占用的 URL，则必须阻止保存并返回显式错误，不允许覆盖目标记录
- 当 URL 冲突导致保存失败时，本次保存不得写出任何 share 正式产物，保持四个产物原子一致

### URL 冲突下的实现与测试要求
- 保存链路必须在生成并写出正式产物前完成 URL 冲突校验
- 测试中必须覆盖：
  - 新建 share 与已有 published URL 冲突
  - 编辑 share 时把 URL 改成另一个已存在 published URL
  - 冲突失败后四个正式产物都不发生变化
- 错误处理只要求给出显式失败结果，不要求本阶段设计额外的高级冲突解决 UI

### 运行时字段来源原则
- `/share` 页面运行时不需要再额外从 `storage.json` 回查 `category` / `folderPath` 来补充列表字段
- `storage.json` 的职责仍是正式存储与保存链路基底，不承担页面过滤时的二次补字段职责
- 若首页继续只消费基础列表能力，则首页可继续只依赖 `list.json` 的现有基础字段，不强制消费目录与分类语义

### URL 修改时必须同步迁移的运行中状态
- 本阶段要求显式迁移的运行中状态仅包括：
  - 当前编辑会话中的 `logoItems` 关联
  - 本次保存所需的旧 URL -> 新 URL 映射
- 不额外引入跨会话缓存迁移、本地持久化迁移或额外客户端状态仓库迁移
- 只要正式产物写回后刷新页面仍能得到正确数据结果，即视为满足本阶段要求

### 刷新后状态要求
- “保存后刷新页面不丢失状态”在本阶段特指：
  - 数据结果不丢失
  - 刷新后页面仍能从正式产物中读取到正确的目录、分类、列表与详情展示结果
- 本阶段**不要求**跨刷新保留目录/分类/搜索/标签的 UI 筛选状态
- 也不要求引入 URL 参数同步或本地持久化来恢复筛选器状态

### 页面与首页依赖关系
- `/share` 页面引入目录/分类能力后，首页仍只依赖 `public/share/list.json` 提供基础分享列表能力
- 本阶段不要求首页同步引入目录树或分类 tabs，只要求首页现有 share 相关展示不回退

### 正式产物字段契约与页面语义的对应关系
- 左侧目录树消费 `folders.json`
- 顶部分类 tabs 候选集合消费 `categories.json`
- 卡片数据与页面过滤直接消费 `list.json`
- 保存链路的完整结构化基底消费 `storage.json`
- 四者必须一起更新，但运行时职责要保持单一，不让页面在多个正式产物之间重复拼字段

### 稳定身份与更新锚点
- 本阶段正式支持 URL 修改，但稳定更新锚点不是 `name`
- 编辑态保存时，更新链路必须显式携带“旧 URL / 当前 URL”，并以旧 URL 作为 published 记录替换锚点
- 对于已有 share：
  - 若当前编辑条目带着旧 URL 保存，则导出层用旧 URL 查找并替换既有 published 记录
  - 若 URL 未变，则直接覆盖同 URL 记录
- `name` 只作为展示字段，不承担记录身份职责
- 新建 share 没有旧 URL，因此仍按“新增 published 记录”处理

### URL 修改时必须同步迁移的运行中状态
- 本阶段要求显式迁移的运行中状态仅包括：
  - 当前编辑会话中的 `logoItems` 关联
  - 本次保存所需的旧 URL -> 新 URL 映射
- 不额外引入跨会话缓存迁移、本地持久化迁移或额外客户端状态仓库迁移
- 只要正式产物写回后刷新页面仍能得到正确数据结果，即视为满足本阶段要求

### 刷新后状态要求
- “保存后刷新页面不丢失状态”在本阶段特指：
  - 数据结果不丢失
  - 刷新后页面仍能从正式产物中读取到正确的目录、分类、列表与详情展示结果
- 本阶段**不要求**跨刷新保留目录/分类/搜索/标签的 UI 筛选状态
- 也不要求引入 URL 参数同步或本地持久化来恢复筛选器状态

### 页面与首页依赖关系
- `/share` 页面引入目录/分类能力后，首页仍只依赖 `public/share/list.json` 提供基础分享列表能力
- 本阶段不要求首页同步引入目录树或分类 tabs，只要求首页现有 share 相关展示不回退

## 保存链路与正式产物规则

### 运行时边界
- `/share` 页面运行时继续消费 `public/share/*` 正式产物
- 不改为页面直接读取 `content.db`
- 不把 web 保存链路强制改成 DB-first

### 本地开发保存
本地开发保存时：
- 读取现有 `public/share/storage.json`
- 基于最新表单状态生成并回写：
  - `public/share/list.json`
  - `public/share/categories.json`
  - `public/share/folders.json`
  - `public/share/storage.json`

### 线上保存
线上保存时：
- 继续走现有 GitHub API 提交链路
- 提交同一组正式产物
- 与本地开发保存保持同一份导出契约

### 正式产物职责
- `list.json`：前台卡片列表直接消费
- `categories.json`：顶部分类 tabs 消费
- `folders.json`：左侧目录树消费
- `storage.json`：正式存储载体，保存更完整结构化信息

### 一致性要求
任何新增、编辑、删除、URL 修改都必须保证四个正式产物一起更新，不能只改 `list.json`。

## 删除与身份语义

### 删除
删除 share 后，必须同步更新：
- `list.json`
- `categories.json`
- `folders.json`
- `storage.json`

### URL 修改
本阶段把“修改 URL”视为正式支持场景。

必须满足：
- 旧 URL 对应记录被正确替换
- logoItems 等运行中关联状态跟着迁移
- 不允许出现旧 URL 与新 URL 同时存在的双 published 记录

### 名称修改
名称变化不应作为主要身份判断依据。

实现与测试应避免继续依赖“通过 name 找旧项”作为核心匹配方式，否则后续会引入错误替换与重复记录风险。

- 分类修改

- `public/share/list.json`

## 风险控制

### 控制范围
本阶段不扩展为“share 版 blog migration 工具工程”。

也就是说：
- 不额外做 share 专属 preview / execute 面板
- 不额外做 share 专属 verify / rebuild 脚本
- 聚焦运行时消费、编辑字段、正式产物一致性与文档边界

### 控制复杂度
- 不让 `src/app/share/page.tsx` 继续膨胀
- 过滤、视图状态与空状态判断要抽到纯逻辑层
- 尽量复用现有 share-storage / share-artifacts 能力
- 目录交互尽量沿用 blog 已验证过的设计模式

### 保持边界稳定
- 页面运行时正式来源不变
- DB 继续留在工具层/迁移层语义，不进入本阶段页面直读路径

## 手动验收

本阶段手动验收采用真实项目运行方式：

- 使用 `pnpm run dev`
- 在浏览器里查看真实页面效果

至少验证：
- `/share` 左侧目录树正常工作
- 顶部分类 tabs 只在当前目录内生效
- 搜索与标签筛选仍可用
- 新建 / 编辑 share 时能设置分类、选择目录、即时新建目录

- 删除 / 改 URL 后不残留脏记录
- 首页依赖 share 列表的位置不回退

## 完成标准

当以下条件全部满足时，认为本阶段完成：
- `/share` 前台真正消费目录与分类正式产物
- 编辑态真正维护 `category` 与 `folderPath`
- 保存 / 删除 / URL 修改会同步更新四个 share 正式产物
- URL 修改不会制造重复 published 记录
- `pnpm run dev` 下真实页面行为稳定
- README / CONTRIBUTING 同步 share 边界说明

## 推荐实施顺序

建议后续计划按以下顺序拆解：
1. 先补失败测试，锁定 `/share` 的目录/分类/过滤语义
2. 抽纯逻辑层，接入双栏导航与分类 tabs
3. 扩展编辑态字段与目录选择/新建能力
4. 收紧保存链路，确保四个正式产物一致更新
5. 做真实页面验证与文档收口
