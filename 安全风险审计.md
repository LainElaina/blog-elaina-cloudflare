# 安全风险审计报告

> 审计时间：2026-03-24
> 项目：blog-elaina-cloudflare（个人博客，Next.js + Cloudflare Workers）

---

## 已修复

### #1 路径穿越 — 任意文件写入（严重）

**文件：** `src/app/api/upload-image/route.ts`

**问题：** `/api/upload-image` 接口接受用户控制的 `path` 参数，直接拼接后写入文件系统，无任何校验。攻击者可以通过 `../../` 穿越路径写入系统任意位置。且该接口无认证，任何人都能调用。

**修复措施：**
1. 限制 `NODE_ENV === 'development'` 才可用，生产环境返回 403
2. 使用 `resolve()` 解析路径后校验必须在 `public/` 目录内
3. 文件类型白名单：仅允许图片（jpg/png/gif/webp/svg/ico/avif）
4. 文件大小限制：10MB
5. 错误信息不再暴露内部细节

## 已知风险（当前架构下可接受）

### #2 加密密钥暴露在客户端

**文件：** `src/consts.ts:15`

`NEXT_PUBLIC_GITHUB_ENCRYPT_KEY` 使用 `NEXT_PUBLIC_` 前缀，会被打包到客户端 JS 中，且有硬编码默认值 `'wudishiduomejimo'`。该密钥用于加密 sessionStorage 中缓存的 PEM。

**为什么可接受：** 此加密是"防明文裸露"而非真正的安全屏障。如果攻击者能执行 JS（XSS），直接读 sessionStorage 即可，加不加密无区别。且需要用户主动开启"缓存PEM"功能。

---

### #3 GitHub Token 明文存 sessionStorage

**文件：** `src/lib/auth.ts:19-26`

GitHub Installation Token 以明文存储在 `sessionStorage` 中。

**为什么可接受：** sessionStorage 受同源策略保护、关闭标签页自动清除。加密存储也无法防御 XSS（攻击者可以直接调用 API 函数）。这是业界常见做法。

---

### #4 Markdown 渲染存在 XSS 可能

**文件：** `src/lib/markdown-renderer.ts:70-71`

Heading 渲染器直接输出 `token.text` 到 HTML，理论上可注入恶意标签。

**为什么可接受：** 博客内容来源于自己的 GitHub 仓库，只有持有私钥的人（即博主本人）能写入。这是 self-XSS，不构成实际攻击面。

---

### #5 自定义组件 iframe 无沙箱

**文件：** `src/components/custom-card.tsx`

`<iframe>` 未设置 `sandbox` 属性，可嵌入任意页面。`imageUrl` 和 `linkUrl` 未校验 URL scheme（如 `javascript:`）。

**为什么可接受：** 自定义组件存在 localStorage，只有本人能创建。唯一外部输入渠道是"导入配置"功能，但需要用户手动选择文件并确认。如需加固，可给 iframe 添加 `sandbox="allow-scripts"` 属性。

---

### #6 API 路由无 CSRF 防护

**文件：** `src/app/api/*`

所有 dev-only API 路由未校验 Origin/Referer，无 CSRF Token。

**为什么可接受：** 这些路由都有 `NODE_ENV === 'development'` 守卫，生产环境不可用。本地开发时监听 localhost，CSRF 攻击需要恶意网页向 localhost 发请求，风险极低。

---

### #7 客户端认证模型

**文件：** `src/lib/auth.ts`、`src/hooks/use-auth.ts`

认证完全在客户端完成（PEM → JWT → GitHub API），服务端 API 无鉴权。

**为什么可接受：** 这是 Jamstack 架构的标准模式。博客部署在 Cloudflare Workers（无状态边缘计算），无传统服务端 session 概念。认证发生在客户端与 GitHub API 之间，博客服务端仅提供静态页面。

---

### #8 `NEXT_PUBLIC_` 暴露配置信息

**文件：** `src/consts.ts:11-15`

GitHub OWNER、REPO、BRANCH、APP_ID 通过 `NEXT_PUBLIC_` 暴露到客户端。

**为什么可接受：** 仓库地址本身是公开信息，APP_ID 可在 GitHub 公开页面查到。唯一敏感的 ENCRYPT_KEY 已在 #2 中说明。

---

### #9 导入配置反序列化不可信 JSON

**文件：** `src/components/import-layout-button.tsx:68-99`

导入功能直接将 JSON 文件内容写入 localStorage 和应用状态，可被注入恶意 iframe/link。

**为什么可接受：** 用户需要手动点击按钮 → 选择文件 → 确认导入，是用户主动行为。风险属于社会工程学攻击范畴，不是代码漏洞。

---

## 最佳实践建议

以下是可选的加固措施，非紧急但有助于提升安全性：

1. **iframe 沙箱**：给自定义组件的 iframe 加 `sandbox="allow-scripts allow-same-origin"`
2. **URL 校验**：自定义组件的 linkUrl 过滤 `javascript:` 协议
3. **导入校验**：导入配置时对 customComponents 的 URL 字段做基本格式校验
4. **密钥轮换**：定期更换 GitHub App 私钥
